Opções de jogos Android

Esta seção descreve as diferentes opções disponíveis para você que controlam como seus projetos de jogo Android serão compilados. As diferentes seções são:

Geral

A primeira parte da seção Geral é dedicada às Informações do Produto e às Configurações de Construção. Você deve preencher as informações do Produto, que incluem o Nome de Exibição do Projeto, suas informações de Pacote, que consiste de um Domínio, Empresa e Produto. As informações da embalagem só podem ser compostas de letras de A-Z e números 0-9, sem símbolos ou outros caracteres especiais, nem devem ter caracteres maiúsculos. Estes detalhes serão então usados para gerar um ID de pacote com o formato "com.company.myappname".

Em seguida, você precisa definir as Ferramentas de Construção, bem como as versões Target, Minimum e Compile SDK.

NOTA: Há também uma opção para a versão "Biblioteca de suporte" a ser utilizada. Do GameMaker v2.2.4 e acima, isto não é mais necessário e quaisquer valores utilizados aqui serão ignorados. A configuração é mantida para uso com projetos legados que requerem um runtime anterior ao 2.2.4.

As Ferramentas de Construção (e Biblioteca de Suporte se necessário) são usadas quando o GameMaker constrói seu jogo para o alvo Android e devem ser definidas para valores apropriados com base nas ferramentas que você tem instaladas usando Android Studio.

  • O SDK alvo indica que você testou seu aplicativo (até e inclusive) na versão que você especifica aqui. Isto é simplesmente para dar ao SO Android uma idéia de como ele deve lidar com seu aplicativo em termos de características do SO. Para todos os fins práticos, a maioria dos aplicativos vai querer definir o Target SDK para a última versão lançada da API (como usada pela configuração do Compile SDK). Isto garantirá que seu aplicativo tenha a melhor aparência possível nos dispositivos mais recentes Android. Note que, a partir de novembro de 2019, o Google fez do nível 28 da API o alvo necessário.
  • A versão SDK mínima é o nível API mínimo que executará seu projeto. A partir de novembro de 2019 o Google fez do nível 16 da API o mínimo absoluto exigido, mas se você adicionar extensions então eles poderão exigir um nível mínimo de API mais alto.
  • A versão Compile SDK é a versão da API contra a qual o projeto é compilado. Isto significa que você pode usar Android recursos da API incluídos nessa versão da API. Se você tentar usar as funcionalidades da API 16 - por exemplo - mas definir o Compile SDK 15, então você receberá um erro de compilação. Entretanto, se você configurar o Compile SDK para 28, ainda poderá executar o aplicativo em um dispositivo API 26 (assim como todas as outras versões anteriores também).

Você pode selecionar a partir de uma das configurações de alvo predefinidas clicando no botão Pré-popular valores SDK para o nível API escolhido, que abrirá uma lista de diferentes APIs para escolher. Selecionando qualquer uma delas, todos os arquivos SDK serão preenchidos com valores apropriados que você poderá então usar ou editar conforme necessário.

NOTA: Você deve ter instalado as APIs apropriadas no Android SDK Manager ou através do Android Studio para que seus projetos possam ser compilados corretamente.

A segunda parte da seção Geral cobre as Arquiteturas de CPU para construir, com as seguintes opções disponíveis (você pode selecionar todas elas ou apenas aquelas que você precisar, mas note que cada uma irá adicionar ao tamanho final do apk compilado):

  • Construir para ARMV5 (verificado por padrão)
  • Construir para ARMV7 (Verificado por padrão)
  • Construir para x86 (verificado por padrão)
  • Build for Mips (verificado por padrão)
  • Construir para ARM64
  • Construir para x86_64

Depois de configurar isso, você pode selecionar as diferentes Orientações nas quais seu jogo pode ser executado para obter a melhor experiência de usuário. As opções disponíveis são:

  • Retrato
  • Portrait-flipped
  • Paisagem
  • Landscape-flipped

Finalmente, você tem as seguintes opções diversas:

  • Habilitar o suporte a Bluetooth/iCade: Isto adicionará suporte para os controles de jogo bluetooth ou iCade ao seu projeto de jogo. Isto está ativado por padrão.
  • Executar análise do código de cotão: Quando esta opção for marcada, as ferramentas de compilação executarão uma análise de código de cotão no projeto à medida que ele for compilando, dando uma verificação de erro mais profunda e mostrando mais informações na janela de saída conforme o projeto compila (um arquivo LOG também será criado, com o path para o arquivo mostrado no final da compilação na Janela de Saída). Note que nem tudo marcado pela análise de cotão como um erro é necessariamente problemático para seu jogo, e ter isto habilitado pode significar que seu jogo não mais será compilado, mesmo que quando estiver desligado, ele compilará e funcionará bem. Essencialmente, isto é apenas para debugging, e a opção está desligada por padrão.
  • Local de instalação: Aqui você pode definir o Local de Instalação padrão para seu jogo. Este pode ser definido como Automático, neste caso o jogo será instalado no local escolhido pelo usuário em seu dispositivo, ou pode ser definido como Preferencial Externo que sempre tentará instalar o jogo no armazenamento externo do dispositivo.

Você também pode definir aqui a Margem de Sono do Android. Esta opção está relacionada à redução da gagueira ao executar seu jogo em sistemas específicos. Basicamente, se seu jogo estiver rodando mais rápido que seu room speed GameMaker "dormirá" pelo tempo restante, mas este sono pode ser bastante impreciso e muitas vezes você pode parar de dormir por mais tempo do que o necessário, fazendo com que seu quadro demore mais, o que causa a gagueira. Para contornar isso, podemos dormir por menos tempo, depois sentar em um loop apertado pelo restante do tempo para torná-lo mais preciso - embora o problema de sentar em um loop seja que isso faz com que o uso da CPU aumente, o que, por sua vez, pode fazer com que a temperatura de sua CPU aumente e seu dispositivo use mais bateria. Por padrão, este valor é ajustado para 4, e em 99,99% dos casos você estará bem com isto, mas para dispositivos de baixa potência ou para dispositivos com muitos processos de fundo rodando, esta pode não ser uma solução ideal e um valor de 5 até 10 pode ser necessário. Observe que esta é uma configuração muito específica do dispositivo e o que funciona em seu dispositivo de construção pode não ser apropriado para outro usuário, e como tal, em caso de dúvida, deixe-o ajustado para 4.

Ícones adaptativos

O Android 8.0 (API nível 26) introduziu ícones de lançador adaptáveis para seus jogos, e estes ícones podem exibir uma variedade de formas através de diferentes modelos de dispositivos. Para lidar com isto, você é obrigado a fornecer uma série de imagens de primeiro plano e de fundo para que seus ícones sejam exibidos em diferentes tamanhos. Estes ícones têm tamanhos diferentes:

  • 81x81 pixels (LDPI)
  • 108x108 pixels MDPI)
  • 162x162 pixels (HDPI)
  • 216x216 pixels (XHDPI)
  • 324x324 pixels (XXHDPI)
  • 432x432 pixels (XXXHDPI)

Você pode fornecer cada imagem forround e de fundo individualmente, ou você pode optar por Gerar Ícones a partir do Arquivo. Isto significa que o GameMaker irá gerar as imagens de ícones necessárias a partir de um arquivo base (XXXHDPI). Você precisa fornecer uma imagem cada uma para o primeiro plano e fundo e então clicar no botão Gerar, e os ícones serão criados para você. As imagens devem estar todas no formato *.png.

É importante entender que seus ícones adaptativos podem ser mascarados em qualquer dispositivo e, portanto, você também tem a opção de ver como eles serão junto com estas máscaras no menu Preview Adaptive Icons:

Note que isto é simplesmente uma prévia de como o ícone poderia parecer com a máscara selecionada, e a configuração não afetará a forma como os ícones são exportados e exibidos, já que isso depende do dispositivo e do SO em que o jogo será executado. Observe também que estes ícones não são criados para você como parte da geração de imagem feita ao usar a ferramenta Project Image Generator.

Gráficos

Aqui você pode alterar os seguintes detalhes relacionados a como seu jogo será exibido, com as seguintes opções disponíveis:

  • Interpolar as cores entre os pixels: Ativa a interpolação linear, que basicamente "suaviza" os pixels. Para gráficos de pixels nítidos, deve ser desligado, mas se você tiver misturas alfa agradáveis e gráficos alisados, é melhor deixá-lo ligado. Isto está ligado por padrão.
  • Profundidade da cor da tela: Isto pode ser usado para definir a profundidade de cor para renderização a 16bit ou 24bit. Isto afetará a compatibilidade com alguns dispositivos mais antigos se for ajustado para 24bit, e também aumentará o ashmem que é necessário (esta é a memória compartilhada nos dispositivos AmazonFire).
  • Suporte de dispositivos: Esta opção definirá se o suporte será somente para dispositivos que tenham uma GPU dedicada, ou todos os dispositivos. Basicamente, se seu jogo tem alguma função avançada de desenho como mistura alfa, mistura aditiva, superfícies, etc... você deve limitá-lo aos dispositivos que têm uma GPU, caso contrário você corre o risco de obter comentários negativos e uma classificação ruim para seu jogo devido a erros gráficos ou falhas em dispositivos mais antigos.
  • Escala: Aqui você pode selecionar a escala da tela de desenho mantendo a relação de aspecto dentro da tela do dispositivo - adicionando "acolchoamento" ao redor das bordas para caber na tela - ou você pode selecionar ter a tela de desenho esticada para caber na tela do dispositivo.

Finalmente, há a opção de definir o tamanho da página de textura. O tamanho padrão (e o mais compatível) é 2048x2048, mas você pode escolher entre 256x256 até 8192x8192. Há também um botão marcado Preview que irá gerar as páginas texture para esta plataforma e depois abrir uma janela para que você possa ver como elas ficam. Isto pode ser muito útil se você desejar ver como as páginas texture estão estruturadas e para evitar ter páginas texture maiores (ou menores) do que o necessário. Para mais informações sobre as páginas texture, por favor, veja aqui.

NOTA: Esteja ciente de que quanto maior o tamanho da página texture, menos compatível seu jogo será com diferentes navegadores e dispositivos.

Ícones

Esta seção permite adicionar as várias imagens de ícones que seu projeto precisará para as diferentes páginas e dispositivos da loja. Todas as imagens de ícones devem ser criadas como arquivos *.png 24bit dos tamanhos apropriados indicados.

Vale notar que o GameMaker possui uma ferramenta Gerador de Imagens de Projeto que pode ser usada para criar automaticamente todas as imagens necessárias para todas as diferentes plataformas alvo para as quais seu jogo está sendo compilado. Se você usar esta ferramenta, você deve revisar as imagens criadas para garantir que elas sejam o que você precisa. A ferramenta gerará ícones assim como imagens (veja a seção abaixo).

Imagens

Aqui você pode definir a Cor da Borda que será usada como fundo para as telas de respingo que não estão definidas para preencher a tela inteira. Você também pode adicionar telas splash ao seu arquivo final do jogo, tanto para os modos paisagem e retrato , que serão mostrados enquanto o jogo carrega no dispositivo. Esta tela deve ser um arquivo *.png de 24 bits, e é recomendável que tenha o mesmo tamanho do primeiro room (ou view) de seu jogo. Se você precisar que a tela de splash seja mostrada por um tempo específico, então você também pode configurá-la aqui de 0 a 10 segundos (o tempo padrão de 0 significa que ela só será mostrada durante a duração do carregamento do asset ). Você também pode definir como a tela de respingo deve ser exibida a partir das seguintes opções:

  • Mantenha a relação de aspecto: Isto manterá a relação de aspecto da imagem em comparação com a tela do dispositivo, adicionando "acolchoamento" ao redor das bordas onde for necessário.
  • Cortar à medida: Isto manterá a imagem 1:1 com a tela do dispositivo e cortará (ou almofada) as bordas, conforme necessário.
  • Esticar: Isto esticará a imagem para preencher a tela do dispositivo (e pode distorcer a imagem mostrada).

Finalmente você pode configurar a Banner Image para o jogo em Android dispositivos de TV, que também é um 24bit *.png.

Embalagem

Uma das características da utilização do Google Play para seus jogos Android é a possibilidade de licenciar seus jogos. Isto significa que seus jogos serão controlados pelo Google para que somente pessoas que tenham feito o download através de seu mercado possam jogar o jogo, evitando assim que cópias ilegais de seu produto sejam distribuídas.

Para ativar esta função em seu jogo, você precisa copiar sua chave pública para a caixa com o rótulo Google Licencing Public Key. Esta licença pode ser encontrada indo à sua página do Google Play Developers e depois selecionando o jogo que você deseja editar. Uma vez na página de perfil do jogo, vá até a seção intitulada "Serviços e APIs" e lá você deverá encontrar a chave pública necessária para que esta funcionalidade seja ativada.

Uma vez ativado, se alguém que não está autorizado a possuir seu jogo tentar jogá-lo, será exibida uma caixa de mensagens pop-up que o informará de que a licença não pode ser encontrada em seu dispositivo e, em seguida, lhes dará a opção de tentar e testar novamente, ou sair do jogo.

Outra característica do Licensing é que seu jogo pode usar APK Expansions. Esta é uma forma de contornar o tamanho máximo de 50MB imposto pelo Google Play (e somente Google Play). Ele criará seu jogo em duas partes: um pequeno arquivo *.apk e um arquivo *.zip maior que conterá todos os seus jogos assets, ambos carregados para a loja. Quando o usuário executar seu jogo pela primeira vez, o zip será baixado e desempacotado. Você pode encontrar mais detalhes sobre esta opção aqui.

Você também pode usar o Google Dynamic Asset Delivery para criar pacotes de aplicativos de até 1GB ao usar a opção de exportação *.aab. Este pacote todo o jogo assets como "install-time" assets que são instalados junto com o próprio aplicativo. Você pode encontrar mais detalhes sobre este recurso aqui.

Finalmente, você tem a opção de preparar o pacote final do aplicativo como um jogo de TV Android (você só deve assinalá-lo se seu jogo estiver de acordo com as diretrizes da TV Android).

Permissões

A partir desta guia você pode alterar algumas das permissões que seu jogo pode solicitar dentro do Manifesto Android. O GameMaker alocará as permissões automaticamente conforme necessário com base nas funções que o jogo utiliza, entretanto você pode achar que precisa de uma permissão para seu jogo que não foi corretamente atribuída (por exemplo, quando você usa o extensions personalizado), neste caso marque a caixa de permissão necessária.

A lista abaixo descreve as permissões que estão disponíveis:

  • WRITE_EXTERNAL_STORAGE: Permitirá que seu jogo escreva para armazenamento externo.
  • READ_PHONE_STATE: Permitir somente o acesso de leitura ao estado telefônico.
  • ACCESS_NETWORK_STATE: Permita que seu jogo tenha acesso a informações sobre redes.
  • INTERNET: Permita que seu jogo abra soquetes de rede.
  • BLUETOOTH: Permitirá que seu jogo se conecte a dispositivos Bluetooth emparelhados.
  • RECORD_AUDIO: Permitirá que seu jogo grave a entrada de áudio do microfone.

NOTA: Se você não tem certeza de que precisa delas, provavelmente deveria deixá-las sem checagem por padrão e deixar o GameMaker lidar com as permissões de seu jogo.

A seção Android Permissões também tem uma área onde você pode injetar permissões nas Tags de Aplicação Android no Androidmanifest.xml. Você pode encontrar uma lista completa de tags disponíveis na documentação Android aqui:

  • Desenvolvedor do Android: Etiquetas de aplicação

Vale notar que para API 28 e acima, você não terá mais permissão para usar qualquer chamada web externa usando "http" - "https", a menos que você adicione a seguinte tag:

android:usesCleartextTraffic="true"

Social

Esta seção é para permitir diferentes características sociais em seu jogo Android. As seguintes opções podem ser escolhidas:

  • Google - Habilitar a economia de nuvens: Ao verificar isso, o Google Cloud Saving permitirá que você faça o download e instale o extension relevante a partir do Marketplace. Veja a seção sobre como salvar nuvens na página Funções Assíncronas.
  • Google - Google Services ID: Isto permite que você insira o ID de Serviços do Google de seu aplicativo.
